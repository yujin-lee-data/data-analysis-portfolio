# 전처리 기준서

> 이 문서는 원본 데이터 처리 원칙, 전처리 기준과 재현 시 유의사항을 정리합니다.

## 1. 전처리 원칙

본 프로젝트에서는 모든 결측치나 차이를 바로 오류로 판단하지 않았습니다. 운영 데이터에서는 주문 상태, 결제 정책, 배송 정책에 따라 정상적으로 비어 있을 수 있는 값도 있기 때문입니다. 따라서 각 항목을 단독으로 보기보다 주문 상태와 날짜 흐름, 결제 금액, 판매자 정보를 함께 확인했습니다.

## 2. 날짜 컬럼 처리 기준

다음 컬럼은 날짜 형식으로 변환합니다.

| 컬럼명 | 의미 |
|---|---|
| order_purchase_timestamp | 주문 발생 시각 |
| order_approved_at | 결제 승인 시각 |
| order_delivered_carrier_date | 배송사 전달 시각 |
| order_delivered_customer_date | 고객 수령 시각 |
| order_estimated_delivery_date | 예상 배송일 |

## 3. 중복 처리 기준

| 항목 | 기준 |
|---|---|
| orders.order_id | 주문 단위 기본 key이므로 중복 여부 확인 필요 |
| payments.order_id | 한 주문에 여러 결제가 있을 수 있으므로 중복 가능 |
| order_items.order_id | 한 주문에 여러 상품이 있을 수 있으므로 중복 가능 |
| sellers.seller_id | 판매자 단위 key이므로 중복 여부 확인 필요 |

## 4. 결측치 처리 기준

| 항목 | 처리 기준 |
|---|---|
| 결제 정보 누락 | 주문 상태와 함께 확인하여 검수 후보로 분류 |
| 승인일 누락 | 주문 상태가 배송 또는 완료 단계인지 함께 확인 |
| 고객 수령일 누락 | order_status가 delivered인 경우 검수 후보로 분류 |
| 리뷰 정보 누락 | 리뷰가 항상 존재하는 것은 아니므로 오류로 보지 않음 |

## 5. 오류 후보 정의 기준

| 오류 코드 | 오류 유형 | 정의 | 해석 |
|---|---|---|---|
| E01 | payment_missing | 주문은 존재하지만 결제 정보가 없는 경우 | 결제/정산 검수 후보 |
| E02 | delivered_without_date | delivered 상태인데 고객 수령일이 없는 경우 | 배송 상태값 검수 후보 |
| E03 | non_delivered_with_date | delivered가 아닌데 고객 수령일이 존재하는 경우 | 상태값 업데이트 검수 후보 |
| E04 | date_reversal | 날짜 순서가 논리적으로 맞지 않는 경우 | 입력 또는 적재 기준 검수 후보 |
| E05 | amount_mismatch | 결제금액과 상품가격+배송비 합계 차이가 큰 경우 | 금액 정합성 검수 후보 |
| E06 | late_delivery | 고객 수령일이 예상 배송일보다 늦은 경우 | 배송 운영 리스크 후보 |

## 6. 날짜 역전 기준

다음 조건 중 하나라도 해당하면 날짜 역전 오류 후보로 분류합니다.

1. 승인일이 주문일보다 빠른 경우
2. 배송사 전달일이 승인일보다 빠른 경우
3. 고객 수령일이 배송사 전달일보다 빠른 경우
4. 고객 수령일이 주문일보다 빠른 경우

## 7. 금액 불일치 기준

주문 상품 가격 합계와 배송비 합계를 더한 값과 결제 금액 합계의 차이가 1 이상인 경우 금액 불일치 후보로 정의합니다.

다만 이 기준은 실제 오류를 확정하는 기준이 아닙니다. 실제 서비스에서는 쿠폰, 포인트, 부분 취소, 환불 등의 정책으로 금액 차이가 발생할 수 있습니다. 따라서 본 프로젝트에서는 금액 불일치를 운영자가 확인해야 할 검수 후보로만 해석합니다.

## 8. 판매자별 오류율 기준

판매자별 오류율은 전체 주문 수가 너무 적은 판매자에게 과대하게 나타날 수 있습니다. 따라서 최소 주문 수 기준을 적용합니다.

본 프로젝트에서는 주문 수가 30건 이상인 판매자를 대상으로 판매자별 오류율을 비교합니다.

## 9. 우선순위 산정 기준

검수 우선순위는 다음 세 가지 기준을 함께 고려합니다.

| 기준 | 의미 |
|---|---|
| 빈도 | 해당 오류 후보가 얼마나 자주 발생하는가 |
| 영향도 | 결제, 배송, 고객 경험, 운영 처리에 얼마나 영향을 줄 수 있는가 |
| 긴급도 | 운영자가 얼마나 빠르게 확인해야 하는가 |

최종 우선순위 점수는 다음과 같이 계산합니다.

```text
priority_score = frequency_score * 0.4 + impact_score * 0.4 + urgency_score * 0.2
```

이 점수는 절대적인 정답이 아니라 제한된 검수 리소스에서 어떤 항목을 먼저 확인할지 정하기 위한 기준입니다.

## 10. 영향도 및 긴급도 점수 부여 기준

본 프로젝트의 impact_score와 urgency_score는 실제 오류 확정이 아니라 운영자가 검수 우선순위를 정하기 위한 기준으로 설정했습니다. 점수는 1~5점 척도로 정의했으며 본 분석에서는 주요 오류 후보가 운영에 미치는 영향이 비교적 큰 항목들이기 때문에 3~5점 범위로 부여했습니다.

| 오류 유형 | impact_score | impact_score 부여 근거 | urgency_score | urgency_score 부여 근거 |
|---|---:|---|---:|---|
| amount_mismatch | 5 | 결제금액과 상품가격+배송비 합계가 맞지 않는 경우로 결제·정산 신뢰도와 직접 연결될 수 있음 | 5 | 정산 전 우선 확인이 필요한 항목으로 판단 |
| payment_missing | 5 | 주문은 존재하지만 결제 정보가 없는 경우로 결제 확인 및 정산 검수에 영향을 줄 수 있음 | 5 | 주문 상태와 결제 정보 연결 여부를 빠르게 확인할 필요가 있음 |
| date_reversal | 4 | 주문일, 승인일, 배송일의 순서가 맞지 않으면 운영 흐름과 상태 해석에 혼선이 생길 수 있음 | 4 | 데이터 적재 또는 상태 업데이트 기준을 확인해야 하는 항목 |
| delivered_without_date | 4 | 배송 완료 상태인데 고객 수령일이 없으면 배송 상태 확인과 고객 안내에 영향을 줄 수 있음 | 4 | 배송 완료 처리 기준을 빠르게 확인할 필요가 있음 |
| non_delivered_with_date | 4 | 배송 완료 상태가 아닌데 고객 수령일이 있으면 상태값 업데이트 기준이 맞지 않을 수 있음 | 4 | 상태값과 배송일 간 불일치 확인 필요 |
| late_delivery | 3 | 배송 지연은 고객 경험에 영향을 줄 수 있지만 결제·정산 오류처럼 즉시 금액 검수로 이어지는 항목은 아님 | 3 | 지속 모니터링이 필요한 항목으로 판단 |

이 점수는 실제 회사의 운영 정책을 반영한 절대 기준이 아니라 공개 데이터 기반 분석에서 검수 우선순위를 정하기 위한 가정입니다. 실제 서비스에서는 CS 발생 여부, 결제 실패 로그, 정산 정책, 내부 SLA 기준을 함께 반영해 점수를 조정해야 합니다.

## 11. 배송 지연과 리뷰 점수 비교 기준

배송 지연 여부에 따른 리뷰 점수 차이는 평균 비교와 Welch t-test 로직을 함께 구성했습니다. 다만 리뷰 점수는 배송 외에도 상품 품질, 고객 응대, 가격, 기대 수준 등 다양한 요인의 영향을 받을 수 있으므로 본 결과는 인과관계가 아니라 고객 경험과 관련된 보조 신호로 해석합니다.

## 12. 금액 불일치 상위 사례 확인 기준

금액 불일치 후보 중 절대 금액 차이가 큰 상위 10건을 `data/processed/amount_mismatch_top10.csv`로 저장합니다. 해당 표는 실제 오류 확정이 아니라 운영자가 추가 확인해야 할 사례 예시로 활용합니다.
